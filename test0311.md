## 通用前置條件

- Laravel 開發伺服器已啟動（`php artisan serve --host=127.0.0.1 --port=8007`）
- 資料庫已 migrate 且有測試資料
- 至少有 **兩個 Organization**（Org-A、Org-B），各自有員工與活動
- Org-A 下有一個 `LotteryEvent`（brand_code 記為 `{BRAND}`）且有設定獎項與可抽員工
- 有一筆 `PrizeWinner` 紀錄含 `claim_token`（用於 P0-5 測試）
- 有一筆 `Payment` 紀錄（用於 P0-4 測試）

## P0-5：Claim Tenant 授權

> 驗證 `/api/claims/verify` 只允許同組織帳號存取

### 前置條件

| 項目 | 說明 |
|------|------|
| User-A | 屬於 Org-A 的使用者，已登入取得 session |
| User-B | 屬於 Org-B 的使用者，已登入取得 session |
| claim_token | Org-A 下某 PrizeWinner 的 64 碼 token |
| prize_id | 該 PrizeWinner 對應的 prize ID |

### 案例 5-1：同組織帳號呼叫 → 成功

| 步驟 | 操作 |
|------|------|
| 1 | 以 **User-A** 身分登入後台 |
| 2 | 發送 `POST /api/claims/verify`，body：`{ "token": "{claim_token}", "prize_id": {prize_id} }` |
| 3 | **預期結果**：HTTP 200，回傳 JSON 含 `employee_name`、`prize_name` 等欄位 |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 5-2：非同組織帳號呼叫 → 404

| 步驟 | 操作 |
|------|------|
| 1 | 以 **User-B**（屬於 Org-B）身分登入後台 |
| 2 | 發送 `POST /api/claims/verify`，body 使用同樣的 `token` 和 `prize_id`（屬於 Org-A） |
| 3 | **預期結果**：HTTP 404，回傳找不到資料（token 不屬於 User-B 的組織） |

## P0-4：ECPay Callback 安全驗證

> 驗證 `/payment/ecpay/notify` 的 MerchantID / TradeAmt / 重複 callback 處理

### 前置條件

| 項目 | 說明 |
|------|------|
| Payment 紀錄 | 一筆狀態為未付款的 Payment，記下 `merchant_trade_no` 和 `amount` |
| MerchantID | `.env` 中設定的 `ECPAY_MERCHANT_ID` 值 |
| CheckMacValue | 可先關閉 MAC 驗證或用工具產生合法簽章（僅測試環境） |

### 案例 4-1：MerchantID 錯誤 → 拒絕

| 步驟 | 操作 |
|------|------|
| 1 | 發送 `POST /payment/ecpay/notify`，body 含正確 `MerchantTradeNo` 但 **MerchantID 設為錯誤值**（如 `9999999`） |
| 2 | **預期結果**：回傳失敗訊息 `MerchantID 不符`，Payment 狀態不變 |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 4-2：TradeAmt 錯誤 → 拒絕

| 步驟 | 操作 |
|------|------|
| 1 | 發送 `POST /payment/ecpay/notify`，body 含正確 `MerchantID` 與 `MerchantTradeNo`，但 **TradeAmt 與 Payment.amount 不符**（如原金額 1000 改為 9999） |
| 2 | **預期結果**：回傳失敗訊息 `TradeAmt 不符`，Payment 狀態不變 |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 4-3：重複 Callback → 冪等處理

| 步驟 | 操作 |
|------|------|
| 1 | 發送一次合法的 `POST /payment/ecpay/notify`（正確 MerchantID、TradeAmt、RtnCode=1） |
| 2 | 確認 Payment 狀態變為已付款 |
| 3 | **再次發送完全相同的請求** |
| 4 | **預期結果**：HTTP 200，回傳 `付款已處理`，不會重複建立紀錄或觸發重複啟用 |

## P0-1：存取碼機制

> 驗證操作員需輸入每日存取碼才能執行抽獎

### 前置條件

| 項目 | 說明 |
|------|------|
| 活動 | Org-A 下的活動（brand_code = `{BRAND}`） |
| 存取碼 | 後台 Filament 面板中顯示的今日存取碼（每日依 HMAC-SHA256 產生，8 碼大寫英數） |
| 無 cookie | 清除瀏覽器 `lottery_access_{BRAND}` cookie |

### 案例 1-1：後台可看到存取碼

| 步驟 | 操作 |
|------|------|
| 1 | 登入 Filament 後台，進入對應的 LotteryEvent 編輯/檢視頁 |
| 2 | 找到「存取碼」或「Access Code」欄位 |
| 3 | **預期結果**：顯示 8 碼大寫英數字串（如 `A7F2C9E1`），每日會變更 |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 1-2：無碼呼叫 draw → 403

| 步驟 | 操作 |
|------|------|
| 1 | 清除 `lottery_access_{BRAND}` cookie |
| 2 | 發送 `POST http://127.0.0.1:8007/{BRAND}/draw`，**不帶** `X-Operator-Token` header |
| 3 | **預期結果**：HTTP 403，回傳 `{ "message": "invalid_operator_token" }` |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 1-3：輸入正確碼 → 驗證通過

| 步驟 | 操作 |
|------|------|
| 1 | 發送 `POST http://127.0.0.1:8007/{BRAND}/verify-access`，body：`{ "access_code": "{今日存取碼}" }` |
| 2 | **預期結果**：HTTP 200，回傳 `{ "success": true, "operatorToken": "..." }` |
| 3 | 檢查 response 的 Set-Cookie，應有 `lottery_access_{BRAND}` cookie（24 小時有效） |

- [ ] **通過** / [ ] 失敗
  備註：__________

### 案例 1-4：頁面載入看到 modal

| 步驟 | 操作 |
|------|------|
| 1 | 清除 `lottery_access_{BRAND}` cookie |
| 2 | 瀏覽器開啟 `http://127.0.0.1:8007/{BRAND}/lottery` |
| 3 | **預期結果**：頁面載入後彈出存取碼輸入 modal，抽獎操作按鈕被禁用 |

|------|------|

| 2 | 按下確認 |
| 3 | **預期結果**：modal 消失，抽獎操作按鈕變為可用，可正常執行抽獎 |
